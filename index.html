New design system

Introducing the first version of Deepak's personal design system — a single source of truth for building consistent, high-quality products.

Why we built this

Every project I build deserves the same quality bar. Instead of reinventing colors, fonts, and components each time, this system codifies every decision once — so future work starts from a strong foundation, not a blank page.

The system is dark-mode first, inspired by Vercel's visual language: clean surfaces, tight typography, and intentional use of color as signal, not decoration.

  • Minor updates add new components or tokens — check release notes
  • Major updates may require migration — a guide is always provided
For developers

Deprecation explained

When a component or token is removed, we follow a 3-stage deprecation process to give you time to migrate.

Process
Deprecated (minor release)
The component is marked deprecated in docs and code. A console warning is shown. It still works.
Migration guide published
A replacement component or approach is documented. At least 2 minor versions before removal.
Removed (next major release)
The deprecated item is removed in the next major version bump.

Color

Color, used strategically and accessibly, reinforces the brand and creates clear and consistent digital experiences. Updated Apr 2026.

Introduction

Color is a core part of the system. It creates cohesive, accessible and balanced visual experiences. The color palette combines brand, neutral and supplemental colors to support a wide range of scenarios, including both light and dark themes and surfaces.

The system includes both primitive colors and semantic collections. Primitive colors serve as base building blocks. Semantic collections are purpose-driven, such as "primary element", and adjust automatically based on context and background.

Primitive colors

Primitive colors are the foundation of the color system. They serve as the core building blocks for creating consistent palettes and scaling the system, but do not carry an assigned purpose.

Primitive colors are grouped into three categories: brand, neutral and supplemental.

What is a surface?

A surface is a special background that controls the color presentation of its child components. In code, a similar process sets backgrounds and configures the surface properties of components placed on top to match.

Surfaces exist at different levels within a layout:

  1. Pages provide the foundational background for all other surfaces, elements and components.
  2. Heroes are usually the most prominent surface on a page, used to introduce the page's content.
  3. Sections divide content into meaningful parts and group related information.
  4. Tiles are contained areas of color that organise smaller pieces of information.

Components, such as Buttons and Notifications, are not surfaces. They aren't directly assigned a surface mode; instead, they inherit and adapt to the surfaces they sit on.

Contrast ratios

All text and interactive elements must meet WCAG 2.1 minimum contrast requirements. The system's semantic tokens are pre-validated against the dark background palette.

PairRatioWCAG AAWCAG AAA
#EDEDED on #000000 18.1:1 ✓ Pass ✓ Pass
#888888 on #000000 4.8:1 ✓ Pass ✕ Fail
#0070F3 on #000000 5.8:1 ✓ Pass ✕ Fail
#00D26A on #000000 8.2:1 ✓ Pass ✓ Pass
#444444 on #000000 2.3:1 ✕ Fail ✕ Fail
⚠ --t3 (#444444) is intentionally low contrast — use only for disabled or decorative text, never for meaningful content.

WCAG guidelines

These rules govern when each contrast level is required based on text size and UI element type.

AA — 4.5:1
Required for normal text (below 18pt / 14pt bold). This is the minimum for most body copy and labels.
AA Large — 3:1
Required for large text (18pt+ / 14pt+ bold) and non-text UI elements like icons and borders.
AAA — 7:1
Enhanced contrast for normal text. Recommended for extended reading, accessibility-critical flows, or users with low vision.
Decorative — exempt
Purely decorative elements, disabled UI, or logotype content are exempt from contrast requirements per WCAG 2.1.

Do's and Don'ts

DO
  • Use --t1 for body text on all dark backgrounds
  • Use semantic feedback tokens for status messages
  • Test color choices with a contrast checker before shipping
  • Supplement color with icons or text labels for colorblind users
DON'T
  • Use --t3 for any meaningful text or interactive labels
  • Rely on color alone to convey status or meaning
  • Use supplemental colors for primary actions or branding
  • Override semantic tokens with raw hex values in components

Writing guidelines

Rules for writing UI copy — labels, buttons, error messages, and empty states — that feel consistent and human.

Buttons and actions
Do
Save changes
Delete account
Get started
Don't
Click here to save
Are you sure?

Error messages
Do
Email address is required.
Password must be at least 8 characters.
Don't
Invalid input.
Error: field validation failed.
General rules
  • Use sentence case, not Title Case, for labels
  • Keep microcopy under 8 words where possible
  • Use active voice — "Save changes" not "Changes will be saved"
  • Avoid technical jargon in error messages
Content

Voice & tone

The design system's voice is consistent — our tone adapts to context. Here's how to get both right.

Voice principles
Clear, not clever
Say exactly what you mean. Avoid puns, jargon, or clever wordplay that might confuse.
Confident, not arrogant
Be direct without being bossy. Guide the user without talking down to them.
Human, not robotic
Write like a person. Use contractions. Sound natural, not formal.
