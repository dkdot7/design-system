Activities and announcements

New design system

Introducing the first version of Deepak's personal design system — a single source of truth for building consistent, high-quality products.

Why we built this

Every project I build deserves the same quality bar. Instead of reinventing colors, fonts, and components each time, this system codifies every decision once — so future work starts from a strong foundation, not a blank page.

The system is dark-mode first, inspired by Vercel's visual language: clean surfaces, tight typography, and intentional use of color as signal, not decoration.

Principles
⬜
Consistent
Every element follows the same rules, always
⚡
Fast
Decisions made once, reused everywhere
◎
Intentional
Nothing is arbitrary — every token has a reason

Using design tokens

Tokens are the single source of truth for all visual decisions. Use them in CSS, JS, or Figma.

In CSS
/* Import once at root */
@import '@deepak/design-system/tokens.css';

/* Use anywhere */
.my-button {
background: var(--t1);
color: var(--bg);
border-radius: var(--r-sm);
}
In JavaScript
import tokens from '@deepak/design-system/tokens.json';

console.log(tokens.color.blue); // "#0070f3"

Introduction to versioning

This design system follows semantic versioning (SemVer). Understanding the version format helps you safely adopt updates.

Version format
1
Major
Breaking changes
0
Minor
New features
0
Patch
Bug fixes only
Upgrade policy
  • Patch updates are always safe to take
  • Minor updates add new components or tokens — check release notes
  • Major updates may require migration — a guide is always provided
For developers

Deprecation explained

When a component or token is removed, we follow a 3-stage deprecation process to give you time to migrate.

Process
Deprecated (minor release)
The component is marked deprecated in docs and code. A console warning is shown. It still works.
Migration guide published
A replacement component or approach is documented. At least 2 minor versions before removal.
Removed (next major release)
The deprecated item is removed in the next major version bump.
For designers

Setup guide

Connect the Figma library to your project to access all components, styles, and variables.

Steps
Open the design system Figma file
Access the shared Figma library link. Duplicate it to your workspace if you need an editable copy.
Enable the library in your project
In Figma, go to Assets → Libraries → find the design system → toggle it on.
Use components from the panel
Open Assets (Cmd+Option+O), search for any component, and drag it onto your canvas.
Swap to use variables
Apply color and text variables from the Variable panel so your designs stay in sync with token updates.
For designers

Design handoff checklist

Before handing off a design to a developer, run through this checklist to ensure nothing is missed.

Foundations
  • All colors use design system tokens, not hardcoded values
  • Typography uses text styles from the library
  • Spacing follows the 4px grid
  • All icons are from the approved icon set
Components and states
  • Default, hover, active, focus, and disabled states designed for interactive elements
  • Error and empty states included for forms
  • Mobile and desktop breakpoints covered
  • Annotations added for spacing, interactions, and edge cases
For designers

Figma Make with VDS

How to use the design system's Figma library to build new features quickly and consistently.

Key principles
Always use library components over creating new ones — detach only when absolutely necessary.
Never hardcode colors. Use variables so designs stay in sync when tokens update.
Workflow
Start from a frame template
Use pre-built page templates from the Figma library to get consistent layouts instantly.
Compose with components
Drag components from Assets and configure via the right panel properties. Override text and icons as needed.
Apply variables, not styles
Use Variable mode (not style picker) for colors so that dark/light mode switching works automatically.
Review against the docs
Cross-reference this site to make sure every component is used as intended — variant, size, and state.

Don't
Invalid input.
Error: field validation failed.
General rules
  • Use sentence case, not Title Case, for labels
  • Keep microcopy under 8 words where possible
  • Use active voice — "Save changes" not "Changes will be saved"
  • Avoid technical jargon in error messages
Content

Voice & tone

The design system's voice is consistent — our tone adapts to context. Here's how to get both right.

Voice principles
Clear, not clever
Say exactly what you mean. Avoid puns, jargon, or clever wordplay that might confuse.
Confident, not arrogant
Be direct without being bossy. Guide the user without talking down to them.
Human, not robotic
Write like a person. Use contractions. Sound natural, not formal.
